# Date and time review criteria

Diff-time checks for temporal correctness — zone handling, storage shape, conversion, clocks, and epoch ranges. This file deliberately gathers the temporal facts other catalogs would scatter — correctness smells and latent failure modes alike, short of the exceptions in Boundaries — because time is one domain and splitting its facts across files invites seam drift. Severity labels (Critical / Important / Suggestion) are defined in [README.md](README.md). If a diff touches no temporal surface — no dates, times, durations, timeouts, schedules, or timestamp storage — this lane reports zero findings; do not manufacture temporal concerns.

Per-stack library recommendations (which date-time library to use on a given stack) are owned by the stack overlays under `overlays/`, not this file.

## Storage and interchange

- **Future local time stored pre-converted** — a user-scheduled future event converted to UTC or epoch at write time, storing only the instant. Zone rules change on government decisions, so the stored instant silently diverges from the wall time the user chose; the tz database's own maintainers state current predictions "will be incorrect after future governments change the rules" ([IANA tz theory](https://data.iana.org/time-zones/theory.html), [Skeet](https://codeblog.jonskeet.uk/2019/03/27/storing-utc-is-not-a-silver-bullet/)). Store future and recurring events as wall time plus IANA zone identifier and re-derive the instant at use. Important; Critical where the schedule drives an external obligation.
- **Offset serialized where a zone is required** — a schema or API carrying only a numeric UTC offset (or bare UTC) for values whose zone rules matter. An offset is not a time zone: it cannot answer "one day later" or survive a DST rule change, which is why [RFC 9557](https://www.rfc-editor.org/rfc/rfc9557.html) extends RFC 3339 to attach an IANA zone name. Important.
- **Recurring events without zone and DST-applicability data** — a recurrence schema storing offset-only or UTC-only times, or lacking a reconciliation path when zone rules change; the W3C guidance is to record the originating zone and whether DST rules apply to future occurrences ([W3C](https://www.w3.org/International/wiki/WorkingWithTimeZones)). Important.
- **Past instant without an explicit offset** — a zone-dependent timestamp serialized without offset in an offset-capable format, or heterogeneous local-time values compared or sorted across sources. Past instants are expressed in UTC (or with an explicit offset) so values compare directly; genuinely floating values (birthdays, holidays) are the exception and are never attached to a zone ([W3C](https://www.w3.org/International/wiki/WorkingWithTimeZones)). Important.
- **`Z` and `+00:00` treated as interchangeable** — serialization code normalizing one to the other. Since [RFC 9557 §2.2](https://www.rfc-editor.org/rfc/rfc9557.html) they differ semantically: `Z` says the local offset is unknown; `+00:00` says UTC is the preferred reference. The instants are equal; the intent metadata is not. Suggestion.
- **Epoch and range bombs** — 32-bit storage of epoch seconds ([signed `time_t` overflows 2038-01-19 and NTP era 0 rolls over 2036-02-07](https://docs.ntpsec.org/latest/rollover.html) — distinct events, [RFC 5905](https://www.rfc-editor.org/rfc/rfc5905)), integer truncation of timestamps in serialization or storage columns, and storage types whose range floor or sub-second precision silently mangles boundary values. Engine-specific type guidance lives in the stack overlays; the reviewable smell here is any new narrow-ranged or truncating temporal storage on a data-bearing path. Important; Critical where the value crosses a persistence or protocol boundary.

## Conversion and arithmetic

- **Ambiguous local time without a disambiguation policy** — converting a local time that occurs twice at a DST fall-back transition with no rule for which occurrence is meant; a field-based local value needs the offset or a DST indicator to disambiguate the repeated hour ([W3C](https://www.w3.org/International/wiki/WorkingWithTimeZones)). Important on scheduling and data-bearing paths.
- **Hardcoded zone rules or fixed offsets for future local times** — DST transition dates baked into code, or a fixed offset applied to a future local time. Zone rules change on short government notice — sometimes with legal effect *before* the tzdata release documenting them — so any hardcoded rule is already wrong somewhere; the [tz-announce list](https://lists.iana.org/hyperkitty/list/tz-announce@iana.org/latest) and [tzdb NEWS](https://data.iana.org/time-zones/tzdb/NEWS) carry the current examples. Important.
- **Pinned tzdata with no update path** — a vendored or container-baked tz database with no mechanism to consume [IANA releases](https://www.iana.org/time-zones); correctness of every zone conversion decays as governments legislate. Important.
- **Elapsed time from wall-clock subtraction** — timeouts, rate limits, or duration math built on subtracting wall-clock readings, which can step backward (NTP corrections, leap seconds) and yield negative durations; this took down [Cloudflare DNS in the 2017 leap second](https://blog.cloudflare.com/how-and-why-the-leap-second-affected-cloudflare-dns/). Use the platform's monotonic clock for elapsed time; [Go's dual-reading `time.Now`](https://go.googlesource.com/proposal/+/master/design/12914-monotonic.md) is the model remedy, and operations that strip the monotonic reading before subtracting reintroduce the bug. Important; Critical in retry, timeout, and rate-limit paths.
- **Sub-second cross-machine clock agreement assumed** — distributed ordering or timestamp comparison that assumes machines agree to better than a second. Leap smearing alone diverges smeared from non-smeared sources by up to ~1 s around a leap event ([Google smear](https://developers.google.com/time/smear), [NTPsec](https://docs.ntpsec.org/latest/leapsmear.html)); ask what ordering guarantee the design actually relies on. Suggestion, framed as a reviewer question — clock topology is rarely visible in a diff.

## Boundaries

- **vs the stack overlays** — this file owns agnostic temporal smells; an overlay owns which library, type, or column type to reach for on its stack.
- **vs `observability.md`** — timestamping and correlation of telemetry belong there; this file owns the correctness of the time values themselves.
- **vs `error-handling.md` / `concurrency.md`** — those own whether a timeout, deadline, or bounded wait exists at all; this file owns whether elapsed time itself is measured correctly (monotonic versus wall clock).
