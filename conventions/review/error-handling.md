# Error-handling review criteria

Diff-time checks for how a change handles failure: result-modeling, exception boundaries, outbound resilience, error context, and the atomicity of interruptible state changes. Severity labels are defined in [README.md](README.md). The result-over-exceptions and timeout-first defaults are owned by `../engineering/architecture-and-design.md` and `../engineering/engineering-philosophy.md`; stack-specific resilience libraries and disposal rules live in the overlays.

- **Results at the domain boundary** — an anticipable domain or business failure returned as an explicit result, not signaled by a thrown exception. Flag a throw used for an expected failure condition in domain or application code.
- **Exception swallowing** — an empty catch, a catch that returns null, or a catch that logs without propagating or converting. Every caught exception is handled meaningfully, converted to a result error, or re-thrown. The stakes are documented: in a study of five production distributed systems, 92% of catastrophic failures resulted from incorrect handling of non-fatal errors the software itself signaled, and over a third traced to handlers as trivially reviewable as an empty block, an overly broad abort, or a TODO left in the handler ([Yuan et al., OSDI '14](https://www.usenix.org/conference/osdi14/technical-sessions/presentation/yuan)).
- **Error context propagation** — errors carry enough context to diagnose without reproducing. Flag messages that drop the original cause, strip identifying data, or read "an error occurred".
- **Resilience on outbound calls** — calls to a network, database, queue, or external API have a retry policy with backoff where appropriate, and a circuit breaker where a down dependency could cascade. A retry policy carries three invariants: backoff is exponential *and randomized* (jitter), attempts are capped per request and bounded by a per-client retry budget (a local ratio each caller tracks — not a coordinated cross-instance counter), and exactly one layer retries a given call path — the layer immediately above the failing call. Retries stacked across layers multiply attempts combinatorially, and unjittered, uncapped retries convert a transient rejection into a self-sustaining outage ([SRE book — cascading failures](https://sre.google/sre-book/addressing-cascading-failures/), [handling overload](https://sre.google/sre-book/handling-overload/)). Flag a retry loop lacking jitter or a cap, and a retry added to a path another layer already retries.
- **Timeout-first** — every outbound call carries a timeout (and a cancellation signal where the platform offers one) from its first implementation. Flag a network or database call with no deadline.
- **Exception-type specificity** — catching the broadest exception type when a narrower one would do. Broad catches are acceptable only at an infrastructure boundary whose intent is explicitly "catch everything, log, continue".
- **Error mapping at layer boundaries** — infrastructure exceptions (storage, transport, file I/O) are caught at the infrastructure layer and converted to domain-meaningful result errors. Domain code does not handle a raw infrastructure exception type.
- **Validation versus domain errors** — input-validation failures are returned as structured validation results at the entry boundary, not as domain errors or exceptions.
- **Idempotency of retryable operations** — anything that can be retried must be safe to run more than once with the same effect as once. Per [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110#section-9.2.2), `GET`/`PUT`/`DELETE` are idempotent but `POST` is not — a create behind a retrying client needs an explicit client-supplied idempotency key, deduplicated server-side. At-least-once messaging guarantees duplicate delivery, so a consumer must dedupe (a natural key, a processed-message record, or an upsert). A multi-side-effect operation with no compensation is not idempotent.
- **Resource disposal and lifetime** — a resource with a disposal contract is released deterministically, synchronous and asynchronous disposal are not mixed up, and a short-lived client is not captured by a long-lived owner. The mechanical cases are often analyzer-caught; review owns the lifetime and ownership judgment.
- **Structured error codes** — errors surfaced to a consumer carry a machine-readable code alongside the human-readable message, so callers can branch programmatically.

## Atomicity

An interruption — crash, kill, redeploy, timeout — can land between any two steps of a state mutation; these bars check that every intermediate state the interruption can expose is one some code path expects. Default severity is Important, rising to Critical on a data-integrity-bearing path. Distributed atomicity stays with its owners: `messaging.md` (dual-write and the outbox), `../engineering/architecture-and-design.md` (aggregates and sagas), `concurrency.md` (in-process shared state).

- **Multi-step mutation without an atomicity mechanism** — a change that writes two or more related records, files, or state locations with no transaction, atomic rename, database constraint, or compensation step spanning them. An interruption between steps leaves partial state that no reader anticipates and no recovery path repairs. Where the writes cross an aggregate, service, or broker boundary, the outbox, saga, and aggregate bars named above own the finding — this bar covers mutations a single mechanism could have covered.
- **Check-then-act as separate steps** — an existence or precondition check followed by a dependent write. Raising the isolation level does not close the race: PostgreSQL documents unique-constraint violations from overlapping Serializable transactions even after the key was checked absent ([PostgreSQL — transaction isolation](https://www.postgresql.org/docs/current/transaction-iso.html)). Use an atomic upsert primitive where the engine offers one, or rely on the constraint itself and handle its violation as the expected concurrent outcome.
- **Isolation-level name treated as a portable guarantee** — code, comments, or configuration mapping an isolation-level name to a specific anomaly protection without naming the target engine. The ANSI definitions fail to characterize real implementations ([Berenson et al., SIGMOD '95](https://arxiv.org/abs/cs/0701157)), and the same name delivers different guarantees across engines — "repeatable read" is snapshot isolation on PostgreSQL but a [consistent-read snapshot that still permits lost updates without locking reads on MySQL/InnoDB](https://dev.mysql.com/doc/refman/8.4/en/innodb-transaction-isolation-levels.html), and Oracle's "serializable" is snapshot isolation, per the [Hermitage](https://github.com/ept/hermitage) empirical test suite. Review against the target engine's documented behavior, never the ANSI table. Suggestion where only a comment or doc mislabels the guarantee; Important where behavior relies on the assumed protection.
- **In-place file rewrite, or an atomic-replace idiom missing its durability steps** — open-truncate-write of an existing data-bearing file, or a write-temp-then-rename that skips an fsync. POSIX requires `rename` to be atomic — a reader sees the old file or the new, never a mix ([POSIX rename](https://pubs.opengroup.org/onlinepubs/9699919799/functions/rename.html)) — but the crash-safe idiom is all four steps: write a temp file on the same filesystem, fsync the file, rename over the target, fsync the containing directory; dropping either fsync reintroduces torn or zero-length outcomes ([LWN — Ensuring data reaches disk](https://lwn.net/Articles/457667/)). Implicit atomicity and ordering assumptions of exactly this kind produced 60 crash-consistency vulnerabilities across 11 mature applications ([Pillai et al., OSDI '14](https://www.usenix.org/conference/osdi14/technical-sessions/presentation/pillai)). Windows has no directory fsync; [`ReplaceFile`](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-replacefilew) is the replacement primitive (it preserves the replaced file's identity, and its `REPLACEFILE_WRITE_THROUGH` flag is documented as not supported), so durability still requires flushing the temp file — [`FlushFileBuffers`](https://learn.microsoft.com/en-us/windows/win32/api/fileapi/nf-fileapi-flushfilebuffers) or a write-through handle — before the replacement.
