# Messaging review criteria

Diff-time checks for asynchronous messaging and event-driven systems — brokers, queues, topics, event streams. The organizing assumption is **at-least-once delivery**: a broker that promises not to lose a message will redeliver it, so every consumer must assume duplicates and every producer must assume its message can arrive more than once, or not at all until acknowledged. That baseline motivates the acknowledgement, outbox, and idempotency bars below. Severity labels are defined in [README.md](README.md).

## Delivery integrity

- **Dual-write without a transactional outbox** — a handler that both persists to a database and publishes to a broker in the same logical operation, with no outbox or message-relay indirection. Publish-then-crash loses the message; publish-before-commit emits a phantom event for state that rolled back. Critical; Suggestion only where the publish target is best-effort telemetry, not a domain or integration event. The design default — an outbox for integration events — is owned by `../engineering/architecture-and-design.md`; this flags the diff-time violation ([Transactional Outbox](https://microservices.io/patterns/data/transactional-outbox.html)).
- **Acknowledging before processing completes** — a consumer that commits the offset or acks the message (or leaves auto-commit on) before its work and side effects have durably completed, so a crash in between silently drops the message. A change claiming exactly-once while performing an external, non-broker side effect is the same defect — end-to-end exactly-once does not extend past broker-to-broker. Critical ([Kafka delivery semantics](https://docs.confluent.io/kafka/design/delivery-semantics.html)).
- **No terminal disposition for a failing message** — a new consumer or subscription whose failure path re-queues or retries with no dead-letter channel, no max-receive or redrive cap, or a `catch` that nacks and requeues forever. Important; Critical where the consumer is ordered or partitioned, so a stuck poison message blocks every message behind it (head-of-line blocking). `error-handling.md` owns whether a retry-and-backoff policy exists; this owns where a repeatedly-failing message finally goes ([Dead Letter Channel](https://www.enterpriseintegrationpatterns.com/patterns/messaging/DeadLetterChannel.html), [SQS redrive](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-dead-letter-queues.html)).

## Ordering

- **Assuming order a partitioned broker does not guarantee** — code that assumes total order across a partitioned or parallel-consumed topic; a producer that omits a partition or ordering key for order-sensitive events, scattering causally related messages; a change that raises consumer concurrency or partition count on an order-dependent flow. Order holds only within a partition, and only for messages sharing a key. Important; Critical where out-of-order application corrupts state (applying an update before the create it depends on, or a stale overwrite). `concurrency.md` owns in-process ordering; this owns the broker's partition-level guarantee and key selection ([Kafka partitions](https://developer.confluent.io/courses/apache-kafka/partitions/)).

## Message contract

- **Missing envelope metadata** — a newly published message emitted as a bare payload with no stable message `id`, explicit `type` and `source`, `specversion`, and `time`. The id (or `source` plus `id`) is the deduplication key a consumer relies on; a message with no stable id cannot be deduplicated. Important; Critical where the missing discriminator forces unsafe consumer guessing on a data-bearing path. The envelope carries fields owned elsewhere — the tenant (`multi-tenancy.md`), correlation and trace context (`observability.md`), and the dedup id (`error-handling.md`) — this bar owns that a standard envelope exists to carry them ([CloudEvents](https://github.com/cloudevents/spec/blob/main/cloudevents/spec.md)).
- **Schema change that breaks the declared compatibility mode** — a change to a schema for an already-published event or topic that violates the subject's registry compatibility mode (backward, forward, full, and their transitive variants), weakens that mode to none, or ignores the producer-then-consumer or consumer-then-producer upgrade order the mode implies. Pair it with the consumer side: a deserializer that rejects unknown fields breaks forward compatibility when a producer adds one — read tolerantly. Important; Critical for a breaking in-place change to a shipped contract. `architecture.md` owns forward-compatible event shape and the breaking-in-place-change flag; this adds only the broker compatibility mode and the tolerant-reader angle ([Schema Registry compatibility](https://docs.confluent.io/platform/current/schema-registry/fundamentals/schema-evolution.html), [Tolerant Reader](https://martinfowler.com/bliki/TolerantReader.html)).
- **Large payload inline** — a producer serializing a large blob (file bytes, an unbounded collection, a base64 document) into the message body instead of storing it externally and passing a reference token. Suggestion for the throughput and cost pressure; Important where it exceeds a hard broker message-size limit and fails the publish at runtime ([Claim-Check](https://learn.microsoft.com/en-us/azure/architecture/patterns/claim-check)).

## Boundaries

- **vs `architecture.md`** — architecture owns forward-compatible event shape, the breaking-in-place-change flag, and expand-and-contract migration. This file adds only the broker compatibility mode and the consumer tolerant-reader angle.
- **vs `error-handling.md`** — error-handling owns idempotent consumers and deduplication, and retry, backoff, and timeout on broker calls. This file owns the delivery-loss direction (acknowledgement timing), the terminal disposition of a failed message, and the dual-write-to-outbox structural gap.
- **vs `concurrency.md`** — concurrency owns in-process ordering and shared-state hazards. This file owns broker partition-level ordering and partition-key selection.
- **vs `multi-tenancy.md`** — multi-tenancy owns that the tenant survives the queue hop and scopes work. This file owns that a standard envelope exists to carry it.
- **vs `observability.md`** — observability owns trace-context propagation across the producer-consumer boundary. This file owns that the envelope carries the correlation and message id in the first place.
