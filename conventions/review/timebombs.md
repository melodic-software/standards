# Timebombs review criteria

Diff-time checks for latent failure modes — changes that pass review and testing at today's volume, age, and lifecycle stage, then fail later under growth, elapsed time, or a restart. Severity labels (Critical / Important / Suggestion) are defined in [README.md](README.md). If a diff touches no growth-capable state, capacity-bearing resource, expiring material, data-lifecycle policy, or process-lifecycle path, this lane is not applicable and reports zero findings; do not manufacture longevity concerns. Where a property is operational rather than visible in the diff, the bar below is phrased as a reviewer question, not an automatic defect.

## Growth

- **Unbounded accumulation** — a change introducing a file, log, cache, table, queue, or in-memory collection that only grows: no rotation, eviction, TTL, retention cap, or compaction, and no stated policy for what happens at the bound. Correct at review-time volume; exhausts disk or memory at production age. Important; Critical where exhaustion takes down a shared host or a data-bearing store.
- **Queue without a shedding policy** — a new or enlarged queue with no rejection or shedding behavior at capacity, or sized far beyond its workers' drain rate. Queued work consumes memory and inflates latency precisely when the system is least able to afford it, delaying load shedding ([SRE book — Addressing Cascading Failures](https://sre.google/sre-book/addressing-cascading-failures/)). The defect is a bound absent or unjustified, not a specific number; sizing calibration (the SRE book's steady-traffic illustration is a queue at half the worker pool or less) is Suggestion-tier. Important. Broker delivery semantics and poison-message disposition stay with `messaging.md`.

## Exhaustion

- **Design against a fixed hard constraint** — a change whose workload can exceed a limit that cannot be raised on request: a serverless runtime ceiling, a message-size ceiling, physical disk, network bandwidth. AWS states plainly that unlike soft quotas "there is no method to change to these limits even under emergency Severity 1 events" ([REL01-BP03](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/rel_manage_service_limits_aware_fixed_limits.html)); hitting one after deployment means degradation until the architecture itself is remediated. Important; Critical where the constraint sits on a data-bearing or availability-critical path.
- **Quota consumed without headroom** — a change provisioning resources or raising API call rates with no check against the governing service quota, or a failover path that ignores that replacement resources count against the same quota while the originals are still counted too ([AWS REL01](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/manage-service-quotas-and-constraints.html), [REL01-BP06](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/rel_manage_service_limits_suff_buffer_limits.html)); a system can pass every normal-operation test and still fail during failover. Important.
- **Capacity math absent** — a change adding per-request writes, per-item disk operations, or per-user in-memory state invites the reviewer question: what is the back-of-the-envelope arithmetic at target scale? The practice non-abstract large system design teaches for iterative design — work the arithmetic before the bottleneck surfaces in production ([SRE workbook — NALSD](https://sre.google/workbook/non-abstract-design/)) — applies just as well at review time. Suggestion, framed as a question — the target scale is rarely in the diff.
- **Identifier or counter near its type ceiling** — a new auto-incrementing key, sequence, or counter on a 32-bit integer type feeding an append-heavy or high-churn surface. The signed 32-bit ceiling (2,147,483,647 — [PostgreSQL numeric types](https://www.postgresql.org/docs/current/datatype-numeric.html)) is reachable within a system's lifetime at modest sustained rates, and the failure arrives as hard insert errors. Important on data-bearing paths. Epoch and timestamp range ceilings are owned by `date-time.md`.

## Expiry

- **Manual renewal in a rotation path** — a certificate, token, or key whose renewal is a manual process, or whose trigger is a notification channel that can silently fail. In the Fortmatic outage the renewal trigger was a single expiry e-mail that a misconfigured group silently blocked, and the certificate expired on schedule ([postmortem](https://medium.com/fortmatic/postmortem-service-disruption-from-expired-ssl-certificate-a993a59272a0)). Automated renewal is the default; a deliberate manual path needs a recorded reason and more than one independent trigger. Important; Critical where the credential fronts an availability-critical surface.
- **Expiry invisible to health monitoring** — status-code health checks treated as covering credential validity. The same postmortem records API logs showing HTTP 200 for the full outage: non-validating clients complete a TLS handshake against an expired certificate, so no alert fired. Dedicated validity monitoring with an expiry threshold is the control; where monitoring topology is not in the diff, ask rather than flag. Important. These two bars are derived from a single first-party incident report, corroborated by monitoring-tooling practice — weight them accordingly.

## Data lifecycle

- **Retention shorter than detection lag** — a retention-window reduction, or new purge or soft-delete logic, whose recovery horizon is shorter than the plausible lag before the data loss is noticed. The slowest-detected losses are low-grade mutation bugs in application code, noticed months after they begin ([SRE book — Data Integrity](https://sre.google/sre-book/data-integrity/)); retention is sized to worst-case detection lag, not to storage cost alone. Important.
- **Replication treated as backup** — a change that removes or weakens backups on the grounds that the data is replicated. Synchronized replicas push a corrupt row or an errant delete to every copy, typically faster than the problem can be isolated ([SRE book — Data Integrity](https://sre.google/sre-book/data-integrity/)). Important; Critical where replication is left as the sole durability story.
- **Restore path changed but never exercised** — a change to backup jobs, recovery tooling, or RTO/RPO-relevant configuration with no corresponding restore exercise, or monitoring that watches backup success rather than restore success. The named anti-pattern is "assuming that a backup exists" ([AWS REL09-BP04](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/rel_backing_up_data_periodic_recovery_testing_data.html)); recovery-from-failure and recovery-from-data-loss are distinct disciplines needing separate exercise ([Google Cloud reliability pillar](https://docs.cloud.google.com/architecture/framework/reliability)). Reviewer question where the exercise cadence is not visible in the diff; Important where the diff itself replaces a tested restore path with an untested one.
- **Pipeline stage timed by offset, not completion** — a multi-stage data pipeline whose later stage runs a fixed interval after the previous one, with correctness depending on the predecessor having finished. As volume grows the predecessor overruns the interval and a growing fraction of data hits the race; Google's deletion-pipeline version of this removed about 600,000 audio references before a user report surfaced it ([SRE book — Data Integrity](https://sre.google/sre-book/data-integrity/)). An explicit completion check replaces the timing assumption. Important; Critical on deletion or irreversible-mutation paths.

## Death and rebirth

- **Recovery that assumes a modest load cut suffices** — recovery logic, autoscaling policy, or runbook automation that assumes dropping load slightly below the pre-incident threshold stops a crash loop. Crash-loop recovery exhibits hysteresis: a service healthy at its old load will keep crashing there once looping, and load must be cut drastically until crashing stops ([SRE book — Addressing Cascading Failures](https://sre.google/sre-book/addressing-cascading-failures/)). Important.
- **Degraded mode without an operator-free exit** — a change adding a degraded or load-shedding mode invites the SRE book's own review question: can the component exit that mode without human intervention ([SRE book — Addressing Cascading Failures](https://sre.google/sre-book/addressing-cascading-failures/))? A degraded mode with no automated exit converts every overload into an operator page. Suggestion, framed as a question — exit behavior is often outside the diff.

## Boundaries

- **vs `date-time.md`** — every temporal failure mode (tzdata churn, DST, epoch and range ceilings, monotonic clocks) is owned by `date-time.md`; this file adds no time facts.
- **vs `error-handling.md`** — error-handling owns not corrupting state while the interruption happens (its [Atomicity cluster](error-handling.md#atomicity)), retry and backoff policy shape, and general resource disposal and lifetime. This file owns what accumulates, exhausts, expires, or fails to come back over time.
- **vs `concurrency.md`** — concurrency-primitive disposal and shared-state hazards live there.
- **vs `messaging.md`** — broker delivery semantics, dead-lettering, and poison-message disposition live there; this file owns growth-boundedness of any queue.
- **vs `multi-tenancy.md`** — multi-tenancy owns per-tenant fairness controls (rate limits, quotas, concurrency caps) that stop one tenant exhausting a shared resource, and the completeness of a tenant's deletion and retention path across stores. This file owns system-wide growth, quota, and retention hazards that are not tenant-attributable.
- **vs `performance.md`** — performance owns the cost of the hot path at today's volume; this file owns what degrades only with growth, age, or lifecycle events.
- **vs `security.md`** — cryptographic strength, secret storage, and trust boundaries live there; this file owns the expiry-and-rotation timing of credentials as a lifecycle hazard.
- **vs `observability.md`** — alert and telemetry design lives there; this file flags only the diff-shaped absence of a validity or completion signal where a bar above requires one.
